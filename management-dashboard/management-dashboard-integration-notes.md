# ملاحظات الربط — لوحة الإدارة (Management Dashboard)

> **للمطوّر:** هذا المستند يشرح كيفية ربط لوحة الإدارة بالبيانات الحقيقية لاحقًا.
> اقرأه بالكامل قبل أي تعديل. اللوحة جاهزة معماريًّا، والربط مطلوب أن يكون **قراءة فقط**.

-----

## 1) طبيعة اللوحة

- لوحة الإدارة **مستقلة تمامًا** عن النظام الأساسي، وتتكوّن من أربعة ملفات فقط:
  - `management-dashboard.html`
  - `management-dashboard.css`
  - `management-dashboard.js`
  - `management-dashboard-sample-data.js`
- هي **نموذج عرض (read-only)** للإدارة: تقرأ البيانات وتحسب المؤشّرات في المتصفح، ولا تكتب أو تعدّل أي شيء.
- حاليًا تعمل على **بيانات تجريبية مُولّدة محليًا** داخل `management-dashboard-sample-data.js` فقط — لا يوجد أي اتصال بقاعدة بيانات.

## 2) قواعد لا يجوز كسرها

- **لا تعدّل النظام الأساسي** (ملفات `index.html` / `quote-builder.html` أو أي منطق إنتاجي). اللوحة إضافة منفصلة.
- **الربط قراءة فقط (READ-ONLY).** لا `INSERT` ولا `UPDATE` ولا `DELETE` من اللوحة.
- لا تغيّر أسماء الملفات الأربعة الحالية.
- المنطق الحسابي كله يبقى في المتصفح كما هو؛ المطلوب منك فقط **تغذية البيانات** ومصدر **دور المستخدم**.

## 3) نقطة الربط الوحيدة للبيانات

كل ملف `management-dashboard.js` يقرأ مصدرًا عالميًّا واحدًا اسمه `window.MGMT_DATA`.
لتفعيل البيانات الحقيقية: **استبدل ملف العيّنة بمُحوّل (adapter)** يجلب البيانات من قاعدة البيانات
(قراءة فقط) ويبني نفس الشكل، ثم يضعه في `window.MGMT_DATA` **قبل** تحميل `management-dashboard.js`.

لا حاجة لتغيير أي شيء آخر في منطق اللوحة.

### الشكل المتوقّع لـ `window.MGMT_DATA`

```
{
  meta:    { product, currency, asOf, rangeStart },
  companies: [ { id, name, created } ],
  staff:     [ { id, name, role } ],          // role: admin | manager | sales
  hotels:    [ { id, name, city, area } ],
  quotations:[ { id, date, companyId, staffId, hotelId, city, region, value, status } ]
}
```

## 4) مصدر البيانات المتوقّع والحقول المطلوبة

البيانات تُقرأ من جدول العروض في النظام الأساسي — على الأرجح **`quotations`** أو **`indo_all_package`**
(يُحدَّد حسب الجدول الفعلي الذي يحوي سجلّات العروض).

الحقول المطلوبة لكل عرض (READ-ONLY):

|الحقل المطلوب|الوصف                   |يُستخدم في اللوحة لـ                                |
|-------------|------------------------|---------------------------------------------------|
|`date`       |تاريخ العرض (YYYY-MM-DD)|كل المؤشّرات الزمنية والرسوم                        |
|`companyName`|اسم الشركة              |صحة الشركات · أفضل الشركات · المفقودة · الخط الزمني|
|`staffName`  |اسم الموظف              |أداء الموظفين · الموظف المسؤول · تنبيهات التركّز    |
|`hotelName`  |اسم الفندق              |الفنادق الأكثر طلبًا                                |
|`destination`|الوجهة/المدينة          |الوجهات الأكثر طلبًا                                |
|`area`       |المنطقة داخل الوجهة     |الوجهات (تُدمج مع `destination`)                    |
|`value`      |قيمة العرض (رقم)        |القيمة · الإيرادات التقديرية · القيمة المفقودة     |
|`status`     |حالة العرض              |معدّل القبول · توزيع الحالات · العروض المعلّقة       |

**قيم `status` المتوقّعة:** `approved` · `sent` · `draft` · `rejected`
(إن اختلفت تسميتها في قاعدة البيانات، حوّلها في المُحوّل إلى هذه القيم الأربع.)

### مطابقة الحقول مع الشكل الداخلي

اللوحة تعمل داخليًّا بمعرّفات (IDs). في المُحوّل، اشتقّ المعرّفات من الأسماء (أو من مفاتيح الجدول):

|من قاعدة البيانات     |إلى `MGMT_DATA`                                |
|----------------------|-----------------------------------------------|
|`companyName`         |`companies[].name` + `quotations[].companyId`  |
|`staffName`           |`staff[].name` + `quotations[].staffId`        |
|`hotelName`           |`hotels[].name` + `quotations[].hotelId`       |
|`destination`         |`quotations[].city` و `hotels[].city`          |
|`destination` + `area`|`quotations[].region` (مثال: «بالي – نوسا دوا»)|
|`value`               |`quotations[].value`                           |
|`status`              |`quotations[].status`                          |
|`date`                |`quotations[].date` و `meta.asOf` (أحدث تاريخ) |


> ملاحظة: حقول `pax` / `nights` غير مطلوبة للوحة الإدارة.

## 5) الصلاحيات وطبقة الدخول (RBAC + Login) — للإدارة فقط

اللوحة **مخصّصة للإدارة فقط**. القاعدة:

- الأدوار: `admin` · `manager` · `sales`.
- **يُسمح بالوصول لـ `admin` و `manager` فقط.**
- **`sales` ممنوع من فتح اللوحة حتى لو عرف الرابط مباشرة** — تظهر له رسالة «هذه الصفحة مخصصة للإدارة فقط».

### بوّابة الدخول الحالية = نموذج تجريبي فقط

أضفنا شاشة دخول بسيطة فوق اللوحة (نوعان: «دخول الإدارة» و«دخول الموظفين») تعمل بكلمات
مرور **تجريبية مكتوبة داخل الكود في الواجهة** — موجودة في `management-dashboard.js`:

```
admin   → admin123
manager → manager123
staff   → staff123
```

> ⚠️ **هذه كلمات مرور نموذج أولي فقط وغير آمنة** (ظاهرة في كود الواجهة). الغرض منها عرض السلوك فقط.
> **عند الدمج الحقيقي يجب حذفها بالكامل** واستبدالها بنظام الدخول الموجود لديك يا بندر.

عند نجاح دخول `admin`/`manager` يُخزَّن الدور مؤقتاً في `sessionStorage` (مع تراجع آمن للذاكرة)،
ويظهر زر **«خروج»** في أعلى اللوحة يُعيد لشاشة الدخول. أي محاولة لفتح اللوحة بدون جلسة صالحة
تُعيد لشاشة الدخول تلقائياً (حماية الرابط المباشر).

### المطلوب منك عند الدمج (نقطة واحدة فقط)

نقطة الربط الوحيدة هي الدالة **`ACCESS.getCurrentRole()`** في `management-dashboard.js`.
يكفي أن تجعلها تُعيد **دور المستخدم الحقيقي** من نظام الدخول لديك:

```js
// داخل ACCESS.getCurrentRole — استبدل قراءة الجلسة التجريبية بنظامك:
getCurrentRole: function () {
  return bandarAuth.getCurrentUser()?.role;   // 'admin' | 'manager' | 'sales'
}
```

ثم **احذف** `PROTOTYPE_PASSWORDS` ودوال بوّابة الدخول التجريبية (`renderLogin` / `checkPrototypeLogin`).
اللوحة تتكفّل بالباقي تلقائياً:

- إن كان الدور `admin` أو `manager` → تُعرض اللوحة كاملة.
- إن كان `sales` (أو أي دور آخر) → تُحجب اللوحة وتظهر رسالة «هذه الصفحة مخصصة للإدارة فقط».
- إن لم يكن هناك مستخدم مسجّل → تُعاد لشاشة الدخول لديك.

> **مهم (أمان):** بوّابة المتصفح للعرض فقط. افرض صلاحية `admin/manager` أيضاً على مستوى
> الخادم / الـ route، فلا تعتمد على الواجهة وحدها لمنع `sales`.

> أداة `?role=` السابقة أُزيلت — صار مصدر الدور هو الجلسة عبر `ACCESS.getCurrentRole()` فقط.

## 6) ملخّص خطوات الربط (READ-ONLY)

1. أنشئ استعلام/View للقراءة فقط على جدول العروض (`quotations` أو `indo_all_package`) يُرجِع الحقول الثمانية أعلاه.
1. اكتب مُحوّلًا يبني `window.MGMT_DATA` بالشكل الموضّح، ويضعه قبل `management-dashboard.js`.
1. اربط `ACCESS.getCurrentRole()` بدور المستخدم الفعلي من نظام الدخول لديك (admin/manager/sales).
1. احذف بوّابة الدخول التجريبية و`PROTOTYPE_PASSWORDS` بالكامل.
1. (مُستحسَن وأساسي) افرض صلاحية `admin/manager` على مستوى الخادم/الـ route أيضاً — لا تكتفِ بالمتصفح.
1. لا تلمس النظام الأساسي ولا أسماء الملفات.
