# مسار الحجوزات (Booking Pipeline) — ملاحظات المعمارية والربط (Phase 10)

> الهدف من هذه المرحلة هو **وضوح سير العمل والمعمارية** قبل بناء الفواتير والأدلة والبرامج.
> نموذج مستقل · لا Supabase · لا تعديل على الإنتاج · بيانات تجريبية فقط.

## 1) كيف تترابط الوحدات (مهم لبندر)

| الوحدة | دورها في دورة الحياة |
| ------ | -------------------- |
| **حالات العروض** (Quotation Status Manager) | يغيّر **حالة العرض**: عرض سعر → مؤكّد/مفقود/ملغى. |
| **الحجوزات المؤكّدة** (Confirmed Bookings Manager) | يدير **سجلات الحجز** الناتجة عن العروض المؤكّدة. |
| **مسار الحجوزات** (Booking Pipeline) | يربط **دورة الحياة التشغيلية** كاملةً ويُظهر المرحلة والإجراء التالي والمسؤول والمتأخرات. |

> **`booking_id` هو المفتاح الرئيسي بعد تأكيد العرض.** كل المستندات المستقبلية تُولّد منه.

## 2) دورة الحياة (٩ مراحل)

```
1 عرض مُنشأ → 2 عرض مؤكّد → 3 حجز مُنشأ → 4 بانتظار تأكيد المورّد →
5 مؤكّد بالكامل → 6 البرنامج النهائي جاهز → 7 الفاتورة جاهزة →
8 دليل الوجهة جاهز → 9 مكتمل
```
المرحلتان 1–2 تحدثان في موديول حالات العروض؛ سجلات الحجز هنا تعيش في المراحل 3–9.

## 3) من سجل الحجز تتفرّع المستندات

```
                       ┌── البرنامج النهائي
                       ├── فاتورة الشركة
   booking_id  ───────►├── فاتورة النقل
                       ├── دليل الوجهة
                       └── تقرير العمليات
```
كل مستند يُولّد لاحقًا من `booking_id` وحقول السجل المرتبطة به.

## 4) حقول سجل المسار (١٩)

```
quotation_id · booking_id · company_name · sales_employee · booking_officer ·
destination · hotel_name · check_in · check_out · booking_status ·
supplier_status · payment_status · final_program_status · invoice_status ·
destination_guide_status · current_stage · next_action · responsible_team · due_date
```

## 5) العرض التشغيلي

- المرحلة الحالية · الإجراء التالي · الفريق المسؤول · تاريخ الاستحقاق.
- طوابير قابلة للتصفية: **عناصر متأخرة · محجوبة على المورّد · بلا برنامج نهائي · بلا فاتورة · بلا دليل وجهة**.
- `current_stage` و`next_action` و`responsible_team` مشتقّة من الحالات الفرعية (يمكن لبندر اعتمادها كما هي أو حسابها على الخادم).

## 6) المستندات المستقبلية (أزرار معطّلة الآن)

أزرار «توليد» معطّلة بصريًّا فقط: البرنامج النهائي · فاتورة الشركة · فاتورة النقل · دليل الوجهة.
لا تُبنى المستندات في هذه المرحلة — تُفعّل لاحقًا لتولّد من `booking_id`.

## 7) علامات الربط المستقبلي

- `SAMPLE DATA SOURCE` · `FUTURE SUPABASE DATA SOURCE` · `DO NOT WRITE IN LAB MODE` (داخل ملف البيانات والمنطق).
- **القراءة:** `window.BP_DATA.pipeline` ← استبدله بـ Supabase view يجمع العرض المؤكّد + الحجز + الحالات الفرعية، بنفس شكل الحقول الـ١٩.
- **الكتابة لاحقًا:** انتقالات المراحل (تحديث الحالات الفرعية) محميّة بالدور (الحجوزات/العمليات/المالية)، مع الحفاظ على `quotation_id` و`sales_employee` (الاحتساب الأصلي) و`booking_officer`.

## 8) الربط بالبوابة (تم في هذه المرحلة)

أُضيف تبويب **«مسار الحجوزات»** إلى بوابة العمليات:
- الإدارة: وصول كامل · فريق الحجوزات: وصول كامل · المبيعات: **لا وصول**.
- يُحمّل `booking-pipeline.html` داخل البوابة (دون دمج)، ويطابق نمط بقية الوحدات.

## 9) القيود (مؤكّدة)

لا Supabase · لا مفاتيح إنتاج · لا تعديل على الإنتاج · لا كتابة فعلية · بيانات تجريبية فقط.
