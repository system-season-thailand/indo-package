# إدارة الحجوزات المؤكّدة — ملاحظات المعمارية والربط (Phase 9)

> نموذج أولي **مستقل** يدير الحجوزات الناتجة عن العروض المؤكّدة.
> **لا Supabase · لا تعديل على النظام · بيانات تجريبية فقط · جاهز للربط لاحقاً.**

## 1) تدفّق العمل

```
عرض سعر  →  مؤكّد (في موديول حالات العروض)  →  سجلّ حجز (هنا)
```
تظهر هنا فقط الحجوزات الناتجة عن عروض **مؤكّدة**.

## 2) الملفات (٥، مستقلة)

`confirmed-bookings-manager.html · .css · .js · confirmed-bookings-manager-sample-data.js · هذا الملف`

## 3) شكل البيانات والحقول

`window.CB_DATA = { meta, statuses[], destinations[], hotels[], companies[], salesEmployees[], bookingOfficers[], bookings[] }`

الحقول المطلوبة لكل حجز:
```
booking_id · quotation_id · company_name · sales_employee · booking_officer ·
destination · hotel_name · check_in · check_out · pax · booking_value ·
booking_status · booking_reference · created_at · notes
```
حالات الحجز: `pending_supplier` · `confirmed` · `partial` · `cancelled` · `completed`.

## 4) الأقسام

1. **الملخّص التنفيذي** — الإجمالي · المؤكّدة · بانتظار المورّد · جزئي · مكتملة · ملغاة · إجمالي القيمة.
2. **قائمة الحجوزات** — بحث + فلاتر (الوجهة · الشركة · موظف الحجوزات · الحالة · نطاق التاريخ). كل صف يفتح تفاصيل الحجز كاملة.
3. **أداء موظفي الحجوزات** — إجمالي الحجوزات · المؤكّدة/المكتملة · نسبة الإلغاء · قيمة الحجوزات.
4. **تحليل الوجهات** — الوجهات الأكثر حجزاً · الفنادق الأكثر حجزاً · متوسط قيمة الحجز.
5. **تحليل عبء العمل اليومي** — حسب اليوم · حسب يوم الأسبوع · حسب الساعة + مؤشرات الذروة (أكثر يوم/يوم أسبوع/ساعة/فترة) لتنظيم المناوبات.

## 5) علامات الربط المستقبلي (للمطوّر بندر)

داخل ملف البيانات والمنطق:
- `SAMPLE DATA SOURCE` — البيانات التجريبية الحالية.
- `FUTURE SUPABASE DATA SOURCE` — استبدلها باستعلام يُرجع صفًّا لكل عرض مؤكّد أصبح حجزاً، بنفس الشكل أعلاه.
- `DO NOT WRITE IN LAB MODE` — هذا الموديول للقراءة والتحليل فقط في المختبر.

**القراءة أولاً:** اربط `window.CB_DATA.bookings` بمصدر القراءة (العروض المؤكّدة + بيانات تشغيل الحجز).
**الكتابة لاحقاً:** تحديث `booking_status` خلال دورة الحجز (بانتظار المورّد → مؤكّد/جزئي/ملغى/مكتمل) سيحتاج صلاحية كتابة محميّة بالدور. حافِظ على نسبة العرض الأصلي (`quotation_id`, `sales_employee`) ومن نفّذ الحجز (`booking_officer`).

## 6) المعمارية المستقبلية — هذا الموديول مصدرٌ لاحقًا لـ:

| الوحدة المستقبلية | الحقول المغذّية لها |
| ----------------- | ------------------- |
| **فواتير النقل** | destination · hotel_name · check_in · check_out · pax |
| **فواتير الشركات** | company_name · booking_value · booking_reference · check_in/out |
| **أدلة الوجهات** | destination · hotel_name |
| **تقارير العمليات** | booking_status · booking_value · booking_officer · created_at |

لذلك صُمّم سجلّ الحجز ليحمل (الشركة · الوجهة · الفندق · التواريخ · المسافرين · القيمة · المرجع · الحالة) — أي وحدة قادمة تقرأ منه دون إعادة تشكيل البيانات.

## 7) الربط بالبوابة

يُضاف لاحقًا كوحدة في بوابة العمليات: عنصر في `nav` (مثال `{ id:"bookings", url:"confirmed-bookings-manager.html", roles:["management","booking"] }`) — وهو يطابق تبويب «الحجوزات المؤكّدة» المحجوز حاليًا في البوابة.

## 8) القيود (مؤكّدة)

لا Supabase · لا مفاتيح إنتاج · لا تعديل على منطق الإنتاج · بيانات تجريبية فقط · بنية جاهزة ليربطها بندر ببيانات حقيقية.
