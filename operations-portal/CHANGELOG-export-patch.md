## No Safari 16-page issue
The 16-page symptom came from iOS Safari's "Share → Print → PDF" path (it ignored
`@page{margin:0}`, inserting a blank after every page → 8 + 8 = 16). The server path renders
with headless **Chromium**, which honors `@page{size:A4;margin:0}` + `preferCSSPageSize`, so the
8 logical `.pdf-page` sheets become exactly 8 PDF pages with no blanks (verified). The Safari
print path is not used by «توليد PDF (خادم)».

## How the button reaches Chromium (unchanged flow)
`#pdfBtn` → preview overlay → **«توليد PDF (خادم)»** → `exportSeasonPDF()` → POST to
`/.netlify/functions/generate-pdf` (or `window.PDF_SERVICE_URL + /export`) → Chromium renders
`pdf-render.html` → PDF downloads. Requires a runtime for Chromium (Netlify Function via Git
deploy, or `pdf-server.js` on a host) — by design; a static drag-and-drop deploy cannot run it.
