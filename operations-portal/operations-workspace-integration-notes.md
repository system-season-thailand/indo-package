# مساحة العمليات (Operations Workspace) — ملاحظات التوحيد والمعمارية (Phase 12)

> **هذه المرحلة توحيد، وليست ميزة جديدة.** الهدف: تقليل تشتّت العمليات بدمج سير العمل في شاشة واحدة.
> نموذج مستقل · لا Supabase · لا تعديل على الإنتاج · لا كتابة فعلية · بيانات تجريبية فقط.

## 1) المبدأ

**سير عمل تشغيلي واحد = شاشة واحدة.** لا يجب أن يتنقّل فريق الحجوزات بين ثلاث وحدات لإدارة حجز واحد.
تتبع كل سجل دورة الحياة كاملةً داخل هذه الشاشة:

```
عرض → مؤكّد → حجز مُنشأ → بانتظار المورّد → مؤكّد بالكامل →
البرنامج النهائي → الفاتورة → دليل الوجهة → مكتمل
```

## 2) الأقسام

1. **المؤشّرات التشغيلية:** النشطة · بانتظار المورّد · المتأخرة · دخول خلال ٧ أيام · مؤكّدة بالكامل · مكتملة.
2. **مسار دورة الحياة:** الشريط الكامل (٩ مراحل) مع عدد السجلات في كل مرحلة؛ المرحلة الحالية ظاهرة دائماً (في الشريط، وفي صف القائمة، وفي مساحة العمل).
3. **إجراءات مطلوبة:** السجلات التي تحتاج تدخّلاً فقط، **مرتّبة حسب الأولوية** (متأخر · دخول وشيك · بانتظار المورّد · بلا برنامج/فاتورة/دليل).
4. **مساحة عمل السجل:** عند الفتح — معلومات تجارية / معلومات الحجز / معلومات تشغيلية / مستندات (أزرار معطّلة) + شريط تقدّم المراحل.
5. **القائمة التشغيلية:** كل السجلات بالأعمدة المطلوبة (رقم العرض · رقم الحجز · الشركة · الوجهة · الفندق · المبيعات · موظف الحجز · المرحلة · المورّد · الدفع · الاستحقاق).

## 3) سلوك الأدوار (يُمرَّر عبر `?role=`)

| الدور | السلوك |
| ----- | ------ |
| الإدارة | رؤية كاملة — **عرض فقط**. |
| الحجوزات | رؤية تشغيلية كاملة (الإجراءات تُفعّل عند الربط). |
| المبيعات | **عرض فقط** لسجلاته هو (تُرشَّح حسب `sales_employee`). |

> في المختبر كل الأزرار معطّلة للجميع؛ الفرق بين الأدوار حالياً هو **نطاق الرؤية** (المبيعات مُقيّد) ووسم الدور.
> عند الربط: تُفعّل إجراءات تغيير المراحل لفريق الحجوزات فقط، وتبقى الإدارة والمبيعات للقراءة.

## 4) توحيد المعمارية (مهم لبندر)

هذه المساحة هي **البيت المستقبلي** لثلاث وحدات حالية، ولا تُكرّر منطقها:

```
مساحة العمليات (Operations Workspace)
├─ حالات العروض      (Quotation Status Manager)
├─ الحجوزات المؤكّدة (Confirmed Bookings Manager)
└─ مسار الحجوزات     (Booking Pipeline)
```

- الوحدات الثلاث **تبقى مؤقتاً** كما هي، ثم تُدمج تدريجياً هنا.
- **مصدر واحد للحقيقة:** سجل واحد يحمل دورة الحياة كاملةً، مفتاحه `booking_id` (و`quotation_id` قبل التأكيد). هذا يمنع تضارب البيانات الذي يحدث حين تملك كل وحدة نسختها.
- خطة الدمج المقترحة:
  1. اجعل هذه المساحة الواجهة الأساسية لفريق الحجوزات.
  2. حوّل «حالات العروض» و«الحجوزات المؤكّدة» إلى **أسطح إجراء** تُفتح من سجل داخل المساحة بدل تبويبات مستقلة.
  3. اعتمد «مسار الحجوزات» كعمود فقري للمراحل داخل المساحة.
  4. عند اكتمال الدمج، أزل التبويبات الثلاثة المنفصلة من البوابة.

## 5) الربط المستقبلي

- `SAMPLE DATA SOURCE` · `FUTURE SUPABASE DATA SOURCE` · `DO NOT WRITE IN LAB MODE` (داخل ملف البيانات والمنطق).
- **القراءة:** استبدل `window.OW_DATA.records` بـ Supabase view يجمع العرض + الحجز + الحالات الفرعية، بنفس شكل الحقول.
- **الكتابة لاحقاً:** انتقالات المراحل محميّة بالدور (الحجوزات/العمليات/المالية)، مع الحفاظ على `quotation_id` و`sales_employee` (الاحتساب) و`booking_officer`.
- المستندات (البرنامج/الفواتير/الدليل) تُولّد لاحقاً من `booking_id`.

## 6) الربط بالبوابة (تم في هذه المرحلة)

أُضيف تبويب **«مساحة العمليات»**: مرئي للإدارة والحجوزات والمبيعات؛ المبيعات يرى سجلاته فقط (`?role=` يُمرَّر تلقائياً).

## 7) القيود (مؤكّدة)

لا Supabase · لا مفاتيح إنتاج · لا تعديل على الإنتاج · لا كتابة فعلية · بيانات تجريبية فقط.
